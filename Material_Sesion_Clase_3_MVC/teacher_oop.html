Material docente (extra)

OOP básico + “¿qué generó Spring Initializr?”

Objetivo: que puedan nombrar el concepto, dar una analogía, y señalar dónde aparece en el proyecto Spring.

Analogías rápidas (para explicar)

  • Clase = receta / plano / formato en blanco.
  • Objeto = plato servido / casa construida / formulario diligenciado.
  • Encapsulación = “no cualquiera manipula el archivo”: se hacen cambios por un procedimiento claro.
  • Abstracción = el control del TV: pocos botones visibles para una lógica interna compleja.
  • Herencia = un “formato base” que se reutiliza y especializa.
  • Polimorfismo = mismo botón “Guardar” pero cada tipo de recurso se guarda de forma distinta.
  • Composición = un recurso contiene autor + categoría + enlace (partes dentro de un todo).

Mini-ejemplo Java (POJO)

Un objeto “modelo” sencillo. Aquí se ve clase/objeto, atributos y métodos.
public class RecursoDidactico {
  private String titulo;
  private String url;
  private boolean activo = true;

  public RecursoDidactico(String titulo, String url) {
    this.titulo = titulo;
    this.url = url;
  }

  public void desactivar() {
    this.activo = false;
  }

  public boolean esValido() {
    return titulo != null && !titulo.isBlank()
        && url != null && url.startsWith("http");
  }
}

// Objeto (instancia)
RecursoDidactico r = new RecursoDidactico("Guía 1", "https://...");
Encapsulación: los atributos están private y el objeto expone acciones con intención (ej: desactivar()).

¿Y esto dónde se usa en Spring?

Concepto OOP En el proyecto (Spring) Cómo lo dirías a estudiantes
Clase/Objeto *Controller, *Service, *Repository son clases; Spring crea los objetos (beans). “El framework construye piezas y las conecta por nosotros”.
Abstracción Un Repository te deja usar save()/findAll() sin saber SQL. “Usamos una interfaz simple; por debajo hay trabajo pesado”.
Encapsulación Separar responsabilidades: Controller no “toca” DB directo; delega a Service/Repository. “Cada clase protege su trabajo: no nos metemos donde no corresponde”.
Polimorfismo Interfaces + múltiples implementaciones (ej: repositorio real vs uno de prueba). “La misma ‘forma’ (métodos) puede tener distintas ‘realizaciones’”.
Composición Entidades con campos/objetos internos; vistas (Thymeleaf) componen una página con fragmentos. “Construimos cosas grandes con partes pequeñas y claras”.
Pregunta guía para el aula
“¿Qué clase crees que debería existir para representar un Recurso didáctico? ¿Qué atributos mínimos tendría y qué método lo validaría?”

Ejemplos de clases (como aparecen en el CRUD)

No memorizar: reconocer patrones y ubicarlos.
// 1) Modelo (OOP) + persistencia (JPA)
@Entity
public class RecursoDidacticoEntity {
  @Id @GeneratedValue
  private Long id;
  private String titulo;
  private String url;
}

// 2) Abstracción: "guardar" sin escribir SQL
public interface RecursoRepository
  extends JpaRepository<RecursoDidacticoEntity, Long> {}

// 3) Polimorfismo + DI: depende de una interfaz
@Service
public class RecursoService {
  private final RecursoRepository repo;
  public RecursoService(RecursoRepository repo) { this.repo = repo; }
}

// 4) Controller: rutas del CRUD (Spring MVC)
@Controller
public class RecursoController {
  private final RecursoService service;
  public RecursoController(RecursoService service) { this.service = service; }
}
Lectura docente: “Spring crea objetos de estas clases y los conecta: eso es inyección de dependencias”.

Analogía completa (para estudiantes)

  • Controller = Secretaría: recibe solicitudes (formularios/rutas) y las envía al área correcta.
  • Service = Coordinación académica: aplica reglas (validar, permitir/denegar).
  • Repository = Archivo/Biblioteca: sabe buscar/guardar registros.
  • H2 = Archivador físico: donde quedan los documentos guardados (persistencia).
  • Thymeleaf = Formato impreso: convierte datos en una vista HTML.
  • Bootstrap = Diseño del formato: que se vea ordenado y usable.
Actividad 2 minutos
Dales una ruta y que digan “qué objeto debería responderla”: GET /recursos → Controller; “traer lista” → Service; “consultar DB” → Repository.

Frase guía

Idea clave
Una clase es un plano. Un objeto es algo construido con ese plano. En Spring, casi todo lo que usamos (Controller, Service, Repository) empieza como una clase.

Mini-actividad rápida

(3–5 min) “Dame un ejemplo de clase y 2 objetos”
  • Clase: Estudiante
  • Objetos: “Ana (grado 10)” y “Luis (grado 11)”
  • Atributos: nombre, grado, activo
  • Métodos: matricular(), cambiarGrado(), desactivar()

Guion sugerido (docente)

  1. Clase = molde/plano (define cómo es algo).
  2. Objeto = instancia real (un “ejemplar” con valores).
  3. Atributos = datos del objeto (estado).
  4. Métodos = acciones del objeto (comportamiento).
  5. Encapsulación = proteger el estado y exponer acciones claras (get/set con propósito).
Conexión con Spring
Un Controller es una clase con métodos que atienden rutas. Spring crea el objeto por ti. Eso se llama inyección de dependencias: Spring “arma” objetos y los conecta.

Mapa rápido de carpetas (lo que deben ubicar hoy):
pom.xml                  # dependencias (Spring Web, Thymeleaf, etc.)
mvnw.cmd                 # Maven Wrapper (Windows)
src/
  main/
    java/...
      *Application.java  # punto de entrada (main)
      ...Controller.java # aquí vivirán endpoints
    resources/
      application.properties  # configuración
      static/                 # CSS/JS/imagenes
      templates/              # HTML Thymeleaf (más adelante)
  test/java/...               # pruebas
Mensaje clave
Si el grupo se pierde: vuelvan a este árbol y localicen 3 cosas: pom.xml, *Application.java, y application.properties.
Dependencias: cómo agregarlas (si faltaron)
En Spring, las “dependencias” no se agregan en el código Java: se agregan en pom.xml. Maven descarga las librerías y luego tu código puede usar anotaciones/clases.
Opción A (más simple): regenerar en Spring Initializr
  • Entra a start.spring.io y usa el mismo Group/Artifact/Java.
  • Agrega la dependencia que faltó (ej: Thymeleaf o JPA).
  • Descarga el ZIP y copia tu carpeta src/ al proyecto nuevo (o compara pom.xml).
Opción B: editar pom.xml (lo real en proyectos)
1) Abre pom.xml y agrega el starter dentro de <dependencies>. Ejemplo:
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-thymeleaf</artifactId>
</dependency>
2) Fuerza descarga/actualización ejecutando (PowerShell, dentro del proyecto):
.\mvnw.cmd -DskipTests package
Si usas IDE: “Reload Maven Project” / “Reimport”.

  • Explorador: abre el árbol y ubica src/main/java y src/main/resources.
  • Buscar (Ctrl+Shift+F): escribe @SpringBootApplication para encontrar la clase principal.
  • Buscar: escribe @RestController para encontrar controladores.
  • Ruta mental: si falla algo, mira el log → copia 1 línea del error → búscala en el proyecto.
Comando estándar (PowerShell)
.\mvnw.cmd spring-boot:run

Tip de didáctica
Si el grupo está tenso: reduce a 2 metas visibles: (1) que corra el servidor, (2) que responda /hello. La estructura y OOP se introducen como “mapa” para orientarse, no para memorizar.
