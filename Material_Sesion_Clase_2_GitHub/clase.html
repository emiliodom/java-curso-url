Sesión de Clase 2 — GitHub y control de versiones

Unidad 2: repositorio, commits, README, .gitignore, push/pull y colaboración mínima.
Abrir ejercicios

Objetivos (hoy)

  • Entender qué problema resuelve Git (historial y trazabilidad).
  • Hacer commits pequeños con mensajes claros.
  • Publicar el repo en GitHub (push) y compartir el enlace.
  • Usar README y .gitignore como práctica profesional.
  • Traducir conceptos a lenguaje didáctico (cómo lo enseñaría en secundaria).

Normas rápidas

  • Un commit = un cambio con propósito.
  • Mensaje con verbo: “Agrega…”, “Corrige…”, “Actualiza…”.
  • Antes de push: revisar git status.
  • Si hay conflicto: no entrar en pánico, leer y resolver con calma.

Guion sugerido (120–140 min)

  • Pregunta detonadora: “¿Cómo demuestro qué cambié y cuándo?”
  • Conecta con docencia: evidencia del aprendizaje, trabajo en equipo y recuperación de versiones.

Analogía rápida
Repositorio = carpeta con historial. Commit = foto con mensaje. Staging = “lo que irá en la foto”. Remoto (GitHub) = copia publicada en internet.
Enfoque docente TIC
Más importante que memorizar comandos: comprender el proceso y usar el historial como evidencia evaluable.
Mini quiz
  • ¿Qué diferencia hay entre guardar y hacer commit?
  • ¿Por qué conviene un commit pequeño?

Usar como carpeta objetivo: Material_Sesion_Clase_1_HTML.

cd Material_Sesion_Clase_1_HTML

git init

git status

git add .

git commit -m "Inicial: material sesión 1"

git log --oneline
Opción VS Code: Source Control → Initialize Repository → Stage All → Commit.

  • Editar/crear README.md con propósito y cómo abrir offline.
  • Crear .gitignore para evitar archivos temporales.
  • Hacer 1–2 commits separados.
Regla
Si el mensaje del commit no explica el cambio, aún no estás listo para commitear.

# Crear repositorio en GitHub (web) y luego:

git branch -M main

git remote add origin https://github.com/USUARIO/REPO.git

git push -u origin main
Opción VS Code: “Publish Branch / Publish to GitHub”.

  • Equipo 3–4: cada integrante hace 1 cambio pequeño (README, texto de ejercicios, estilos).
  • Por cada cambio: commit claro y push.
  • Registrar link del repo + capturas del historial.
Meta
Que el historial cuente la historia del proyecto.

  • Ticket: 1 cosa que versionarías siempre y 1 duda.
  • Pregunta didáctica: “¿cómo evaluaría el proceso usando commits como evidencia?”
