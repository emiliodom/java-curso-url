Guía de ejercicios

Ejercicios de Git y GitHub (paso a paso)

Meta: repositorio en GitHub con 3+ commits + README.
Ir al entregable
Modalidad: GitHub Codespaces (importante)
Estas prácticas están diseñadas para hacerse en GitHub Codespaces, así que no dependen de carpetas locales del docente. Trabajarás en un repositorio nuevo (en tu cuenta o en el espacio de práctica de la clase).
Enfoque docente TIC: en cada ejercicio, piensa cómo convertir el paso técnico en una regla didáctica simple (“siempre haz…”, “nunca hagas…”).
Nombre recomendado del repo
sesion-2-git-github-practica (o similar). Evita espacios y usa guiones.

Ejercicio 1 — Crear repo + abrir en Codespaces

Objetivo: tener un entorno listo para trabajar (repo remoto + editor + terminal).

  1. En GitHub, crea un repositorio nuevo (puede ser público o privado).
  2. Activa: Add a README (para que exista un commit inicial).
  3. Abre el repo y haz clic en Code → Codespaces → Create codespace.
  4. En el Codespace, abre la terminal y ejecuta git status.
Resultado esperado
Estás dentro de un Codespace, el repo está clonado, y git status dice algo como: On branch main y nothing to commit, working tree clean.
Hints completos
  • Si no ves “Codespaces”, revisa permisos o usa la cuenta institucional.
  • Si el repo está vacío, crea el README desde la web (primer commit) y recarga el Codespace.
  • Regla docente: “Si no hay commit inicial, no hay historia”.

Ejercicio 2 — “Producto mínimo” + commits pequeños

Objetivo: practicar el ciclo editar → stage → commit → verificar historial.

  1. Crea un archivo index.html con un título y un párrafo (tu “mini-sitio”).
  2. Crea styles.css con un estilo simple (color y tipografía).
  3. Haz 2 commits separados: uno para HTML y otro para CSS.
  4. Revisa el historial con git log --oneline.
Comandos (si los necesitas) 
# Crear archivos
touch index.html styles.css

# Ver cambios
git status

# Commit 1 (HTML)
git add index.html
git commit -m "feat: agrega estructura HTML inicial"

# Commit 2 (CSS)
git add styles.css
git commit -m "style: agrega estilos base"

git log --oneline
Resultado esperado
En git log --oneline ves (al menos) 2 commits nuevos y mensajes entendibles.
Hints didácticos
  • “feat:” = agregaste algo visible; “fix:” = arreglaste un problema; “docs:” = documentación; “style:” = apariencia.
  • Regla docente: “Un commit = una idea”. Si cuesta explicar el commit, sepáralo.

Ejercicio 3 — README docente (explica el proyecto)

Objetivo: documentar para que otra persona (estudiante) use el repo sin ayuda.

  1. Edita README.md y agrega secciones: Propósito, Cómo ejecutar/ver, Estructura.
  2. Incluye una frase de evaluación formativa: ¿qué evidencia aporta el historial?
  3. Haz commit con prefijo docs:.
Plantilla lista para copiar 
# Sesión 2 — Práctica Git/GitHub

## Propósito
Este repositorio practica control de versiones para uso educativo.

## Cómo ver el resultado
- Abre `index.html` (o usa la previsualización del editor).

## Estructura
- `index.html`: contenido
- `styles.css`: estilos

## Evidencia del proceso (docencia TIC)
- Los commits muestran el avance paso a paso (qué cambió y por qué).
Resultado esperado
README.md se ve bien en GitHub (se renderiza) y explica el repo en lenguaje claro.
Hint (mensaje de commit)
docs: agrega README con propósito e instrucciones

Ejercicio 4 — .gitignore + “higiene” de repositorio

Objetivo: practicar una buena costumbre docente/profesional.

  1. Crea .gitignore.
  2. Agrega reglas típicas (SO / editor / logs).
  3. Commit con prefijo chore: o build: (lo importante es que sea entendible).
Plantilla mínima 
# OS
.DS_Store
Thumbs.db

# VS Code
.vscode/

# Logs
*.log
Resultado esperado
git status queda limpio y el repositorio no sube “basura”.

Ejercicio 5 — Ramas: crea una rama y súbela

Objetivo: entender que los cambios van en ramas y se integran con PR.

  1. Crea una rama llamada docs/guia-uso.
  2. En esa rama, mejora el README (agrega una sección “Cómo evaluar con commits”).
  3. Commit y push de la rama.
Comandos (alternativa) 
git switch -c docs/guia-uso

# editar README.md

git add README.md
git commit -m "docs: agrega pauta de evaluación con commits"

git push -u origin docs/guia-uso
Resultado esperado
En GitHub aparece la rama docs/guia-uso con tus commits.
Hint docente
Regla: “main es lo estable; lo nuevo se trabaja en ramas”.

Ejercicio 6 — Pull Request (PR): integra tu rama a main

Objetivo: practicar el flujo profesional (rama → PR → revisión → merge).

  1. En GitHub, abre un Pull Request desde docs/guia-uso hacia main.
  2. En la descripción del PR, explica: qué cambiaste y por qué.
  3. Simula revisión: agrega un comentario (puede ser tuyo) tipo “¿podemos simplificar esta frase para estudiantes?”
  4. Haz un ajuste pequeño en la rama, commitea y vuelve a push (para ver el PR actualizarse).
  5. Haz Merge del PR (recomendado: Squash and merge para mantener historia limpia).
  6. En el Codespace: vuelve a main y haz git pull.
Resultado esperado
El PR queda en estado Merged, y tu rama se integra a main. En local, git log --oneline muestra el commit del merge/squash.
Hints completos (si te pierdes)
  • Un PR es una “solicitud de integrar cambios”. Es ideal para enseñar revisión y argumentación.
  • Si git pull no trae nada, revisa que estés en main (git branch).
  • Regla docente: “La revisión es parte del aprendizaje, no castigo”.

Reto final (25–35 min) — Equipo: ramas + PR como evidencia

Forma equipos de 3–4. Objetivo: un repositorio del equipo con ramas por integrante y al menos 2 Pull Requests. Usen la plantilla de entregable.

  1. Definan un repositorio del equipo (uno solo) y abran un Codespace.
  2. Asignen roles: Driver/Navigator/Tester/Documentador (rotación rápida).
  3. Cada integrante crea una rama: feat/nombre-navbar, docs/nombre-readme, etc.
  4. Cada rama debe tener 1–2 commits pequeños con mensajes claros.
  5. Abren PRs, hacen al menos 1 comentario de revisión y hacen merge.
  6. Registra links: repo + PR(s) + evidencia (capturas o enlaces).
  7. Reflexión: ¿cómo usarías ramas/PR como evidencia y para enseñar colaboración?
Regla de oro
Si dos personas editan lo mismo al mismo tiempo, habrá conflictos. Solución docente: turnos, ramas y PR con revisión (no “pegar” cambios a la fuerza).
Mini cierre didáctico
Redacten 3 normas para su aula: (1) nombres de ramas, (2) mensajes de commit, (3) cuándo usar PR y revisión.

Mini-rúbrica sugerida (enfoque docente TIC)

Úsenla como ejemplo y ajústenla a su contexto. Enfatiza proceso y evidencia.

Criterio Puntos Qué se observa
Repositorio publicado 25 Link funciona, se ven archivos y README.
Historial como evidencia 35 3+ commits, mensajes claros, commits pequeños (no “todo en uno”).
Documentación (README/.gitignore) 25 README entendible para estudiantes; .gitignore evita temporales.
Reflexión pedagógica 15 Explica cómo enseñaría y evaluaría el proceso; menciona ética.
Ejemplo de criterios formativos (basados en commits)
  • “Cada commit tiene un propósito claro (verbo + objeto)”.
  • “Puedo leer el historial y entender la historia del trabajo”.
  • “El equipo no mezcla cambios no relacionados en un mismo commit”.
  • “El README permite que otro abra el material sin ayuda del docente”.
